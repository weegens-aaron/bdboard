# 0007 — UI-triggered formula pour (a workspace-mutating, work-spawning action via `bd mol pour`)

- **Status:** accepted
- **Date:** 2026-06-02
- **Relates to:** bdboard-ain (epic, "Add beads to the board from a formula via
  a UI element"), bdboard-9n4 (spike,
  `docs/design/bdboard-9n4/formula-to-board-ui-spike.md`), bdboard-obv (feature
  catalog, `docs/catalog/pour-formula.md`), bdboard-fh8p / bdboard-ksny /
  bdboard-078p (UI hardening). Builds on ADR 0004 (CLI is the source of truth),
  ADR 0005 (live-refresh pipeline), and ADR 0006 (manual field-editing model).
- **Supersedes / Superseded by:** none.

> bdboard exposes a UI action that **mutates the workspace by creating whole
> bead trees** — it shells out to `bd mol pour <formula> --var k=v … --json`,
> reuses the existing serialized-mutate / CSRF / SSE / cache plumbing, and
> renames the molecule wrapper to `<formula> <short-id>`. This is a deliberate,
> bounded expansion of bdboard's write surface beyond the field-value edits of
> ADR 0006, and it is the second write class allowed past the "read-only
> observer" framing of ADRs 0004/0005.

## Context

ADRs 0004 ("bdboard never writes to `.beads/`") and 0005 ("one-way observer
chain") describe bdboard as a read-mostly dashboard. ADR 0006 then carved out a
**narrow** first write surface: editing the *values of fields that already
exist* on a bead, and it explicitly left "shape/graph edits" and "creating new
beads" out of scope.

The bdboard-ain epic asked for something categorically larger: let a human
**pour a formula from the dashboard**, materializing an entire molecule tree
(wrapper + epic root + child tasks) onto the board live. Spike bdboard-9n4
verified, empirically against the installed `bd`, exactly how to do this and
surfaced the sharp edges:

- The pour mechanism is `bd mol pour <name> --var k=v … --json` — it **cooks the
  formula inline** (no separate `bd cook`), returns `new_epic_id` /
  `id_mapping` / `created`, and **mutates** the workspace.
- A pour is **atomic**: a failed pour rolls back to zero new beads, so the board
  is never left with an orphaned partial tree.
- Variable enumeration must read the on-disk `*.formula.json` (path from
  `bd formula list --json` → `source`), because `bd formula show --json` omits
  the `variables` block and `formula list`'s `vars` count is wrong (always 0).
- The grouping node `bd mol pour` creates is `issue_type == "molecule"`, not
  `epic`, so it does not appear in the epic strip without intervention; bdboard
  renames a node to `<formula> <short-id>` so repeated pours are
  distinguishable.

This is a durable, cross-cutting decision: a contributor reading only the code +
ADRs 0004/0005 would (correctly) believe bdboard is read-only and could try to
"simplify away" the pour write path as a contradiction. The decision — and the
boundary that keeps it safe — needs a canonical home.

## Decision

- **bdboard mutates the workspace through a second guarded affordance: formula
  pour.** The write goes through the `bd` CLI (`bd mol pour … --json`), never by
  touching `.beads/` files directly — this honors the ADR 0004 invariant
  ("the CLI is the source of truth; bd owns validation and locking"). The
  "never writes to `.beads/`" language in ADRs 0004/0005 means **never writes
  the files directly** and **never originates the read-feedback loop**, not
  "never mutates state via bd."
- **One JSON-returning hybrid mutation,** `BdClient.pour_formula(name,
  variables)`, runs under the existing `_sub` (bd's embedded dolt is
  single-writer), surfaces bd's stderr on failure, parses the JSON result, and
  invalidates caches — mirroring the `update_field` / `remember` patterns rather
  than adding new infrastructure (DRY with ADR 0006).
- **Reads, not writes, for discovery.** Formula listing
  (`BdClient.list_formulas` → `bd formula list --json`) and variable/step
  enumeration (`read_formula_variables` / `read_formula_detail`) parse the
  static `*.formula.json` template the CLI points us at. Reading that template
  is **consistent** with ADR 0004: we still discover *via the CLI* (it hands us
  the `source` path), and the template is a static file, not the issue store —
  bdboard still never reads `.beads/issues.jsonl`.
- **Live refresh reuses ADR 0005's pipeline.** A successful pour lands on every
  board via the existing watcher → `Store.refresh` → SSE `beads_changed` chain;
  the pour route additionally fires an **optimistic** `beads_changed` broadcast
  so the acting tab updates immediately without waiting on the watch debounce
  (same pattern as the memory-create route).
- **Naming is bdboard's job, not the formula's.** After a pour, bdboard renames
  the molecule wrapper (the returned `new_epic_id`) to `<formula> <short-id>` so
  repeated pours of the same static template are distinguishable on the board.
  The unique id is generated by bdboard wrapping the pour, never baked into the
  reusable formula template.
- **Safety relies on pour atomicity + stderr surfacing, not `--dry-run`.**
  `--dry-run` does **not** exercise dependency creation (it cannot catch the
  task→epic blocks-edge failure the spike found), so the UI must surface the
  *real* pour's stderr. Atomic rollback guarantees a failed pour leaves zero new
  beads, so the board never shows a partial tree.

## Alternatives considered

| Option | Pros | Cons | Verdict |
|---|---|---|---|
| **A. UI pour via `bd mol pour … --json` on the existing mutate/CSRF/SSE/cache plumbing; discovery by reading the CLI-pointed `*.formula.json`** | Reuses 100% of existing write/CSRF/SSE/cache code; preserves the CLI-is-source-of-truth invariant; atomic rollback makes it board-safe; one new method | A second write surface to reason about; depends on bd's pour JSON shape | **CHOSEN** |
| B. Keep bdboard strictly read-only; pour only from the terminal | Zero new write surface; ADRs 0004/0005 stay literally true | Defeats the entire bdboard-ain epic; the at-a-glance dashboard can't spawn work where the user already is | Rejected |
| C. Two-step `bd cook` (persist proto) then `bd pour` | Mirrors a mental "stage then apply" model | Unnecessary — `mol pour` cooks inline; adds a second subprocess + a transient proto to clean up | Rejected |
| D. Drive the variable form from `bd formula show --json` / the `vars` count | Pure-CLI, no template file read | `formula show --json` omits `variables`; the `vars` count is always 0 — both are empirically wrong (spike bdboard-9n4) | Rejected |
| E. Write the pour by editing `.beads/` / Dolt directly | No CLI round-trip | Violates ADR 0004; bypasses bd validation, locking, and atomic rollback | Rejected |

## Consequences

- **bdboard now has two documented write surfaces:** field-value edits
  (ADR 0006) and formula pours (this ADR). Any *third* write class (e.g. a
  status-lifecycle affordance, still deferred by ADR 0006) needs its own ADR or
  an amendment here — the "read-only observer" framing of ADRs 0004/0005 must be
  read as "no direct file writes / no read-loop origination," not "no
  mutation."
- **The CLI-is-source-of-truth invariant (ADR 0004) is preserved:** every
  mutation, including pour, goes through `bd`; bdboard still never writes
  `.beads/` files nor reads `issues.jsonl`.
- **bdboard is coupled to bd's `mol pour --json` contract** (`new_epic_id`,
  `id_mapping`, `created`) and to the `*.formula.json` on-disk schema
  (`variables` map). If a future bd release adds `variables` to
  `formula show --json`, the template read can be dropped (discovery moves fully
  behind the CLI) — a documented simplification, not a required one.
- **Repeated pours of one static formula are safe and distinguishable:** atomic
  rollback prevents partial trees; the `<formula> <short-id>` rename keeps the
  board readable.
- **What would make us revisit:** bd changing the pour JSON shape or the formula
  template schema; demand for non-atomic / streaming pours; or a decision to
  generalize the write surface (at which point this and ADR 0006 should be
  reconciled into a single "bdboard write model" ADR).
